01 Avaliação Técnica — O Que Está Certo e o Que Falta Diagnóstico honesto antes de arquitetar
Veredicto Geral

A visão do produto é genuinamente sólida. A interface conversacional PWA resolve um problema real melhor do que qualquer alternativa existente no segmento. O modelo de negócio é sustentável. O stack técnico escolhido (Laravel + Vanilla JS) é a decisão certa para o contexto. O projeto está pronto para ser construído — com as lacunas abaixo endereçadas antes de escrever a primeira linha de código.

Alerta Crítico — LINE Notify

O LINE Notify foi oficialmente descontinuado em março de 2024. Qualquer integração usando esta API irá falhar em produção. A solução correta para o Japão é a LINE Messaging API com um canal de notificação dedicado. Isso exige a criação de um LINE Official Account do sistema (não do lojista) para envio de alertas. Custo: gratuito até 200 mensagens/mês por conta.

Decisão Arquitetural — Multi-Tenancy

Usar Shared Database com tenant_id em todas as tabelas, com Global Scope no Eloquent aplicando filtro automático por tenant. É a abordagem correta para o estágio atual: simples de implementar, performática até ~10.000 tenants, e migra facilmente para schemas separados se necessário. Cada tenant é identificado pelo UUID do estabelecimento, resolvido via middleware antes de qualquer query.

07 Stack Técnico Definitivo e Justificativas
Backend Framework
Laravel 11 + PHP 8.3
Eloquent com Global Scopes resolve multi-tenancy elegantemente. Queues nativas para jobs assíncronos. Sanctum para API auth. Política de segurança madura.
Banco de Dados
MySQL 8.0
Transações ACID para integridade. JSON columns para bot_scripts. Índices compostos em (tenant_id, created_at) para queries de dashboard. Disponível em Hostinger.
Cache e Filas
Redis (ou database driver)
Cache de bot_scripts evita query a cada scan de QR. Filas para envio assíncrono de notificações. Se Hostinger não tiver Redis, database driver funciona até 500 tenants.
Frontend — PWA
Vanilla JS + CSS + Service Worker
Zero dependências = carregamento <50kb = <1 segundo em 3G. Sem React, sem Vue — JavaScript puro é suficiente e ideal para contexto mobile-first.
Frontend — Painel
Laravel Blade + Alpine.js
Alpine.js para interatividade leve (filtros, modais, preview do bot) sem overhead de SPA. Blade components para reutilização. Livewire se necessário para tempo real.
Notificações BR
Evolution API + fallback SMTP
Evolution API é a opção mais acessível para WhatsApp unofficial. Fallback obrigatório: se falhar 3 tentativas, envia e-mail. Meta Cloud API quando crescer.
Notificações JP
LINE Messaging API (não LINE Notify)
LINE Notify foi descontinuado. Messaging API com Webhook é o caminho oficial. Um LINE Official Account do sistema envia alertas para os donos cadastrados.
E-mail
Mailgun ou SMTP Hostinger
Laravel Mail com templates Blade em PT e JA. Relatórios mensais em HTML responsivo. SMTP do Hostinger funciona para início, Mailgun para escala.
Upload / Storage
Laravel Storage (local → S3)
Fotos de feedback salvas localmente no Hostinger inicialmente. Migration para S3/R2 quando necessário. Validação: MIME, tamanho, extensão no servidor.
Autenticação
Laravel Sanctum + Policies
Sanctum para tokens de API (PWA → backend). Session auth para painel web. Policies para isolar dados por tenant (Gate::before para super_admin).
QR Code
simplesoftwarebv/simple-qrcode
Biblioteca PHP nativa, sem dependência externa. Gera SVG e PNG. Logo do tenant sobreposto via GD ou Imagick. Export PDF via dompdf.
Hospedagem
Hostinger Premium (atual) → Cloud
Hostinger Premium suporta até ~50 tenants confortavelmente. Migrar para Hostinger Cloud KVM ao passar de 50 para ter SSH, Redis nativo e mais controle.
08 Segurança — Checklist Completo Não opcional — implementar desde o primeiro dia
🔐 API (endpoint público /api/review)
Rate limiting: 3 reviews por IP por hora via throttle middleware
Validação de slug: tenant deve existir e estar ativo antes de aceitar dados
PreparedStatements via Eloquent em todas as queries (padrão Laravel)
Sanitização de texto: strip_tags + mb_substr antes de persistir
Upload de foto: validação MIME no servidor (não só extensão), máx 5MB, rename para UUID
CORS restrito ao domínio do PWA — sem wildcard em produção
🏠 Painel (autenticado)
CSRF token em todos os formulários (Laravel padrão)
Policies garantem que lojista só acessa dados do próprio tenant_id
Super admin nunca vazado no token — role verificada no servidor
Sessão expira em 2 horas de inatividade
2FA opcional para super admin (Google Authenticator)
Audit log: toda ação admin registrada com IP, user e timestamp
📊 Dados do Consumidor
Contato do cliente (e-mail) armazenado criptografado (encrypt() do Laravel)
Session token anônimo para closing the loop — sem PII vinculada
Fotos armazenadas fora da pasta public — servidas via signed URL
Nenhum cookie de tracking no PWA — sem Google Analytics
LGPD/PDPA: aviso de privacidade na tela inicial do PWA
🔔 Notificações Externas
Credenciais WhatsApp/LINE em .env — nunca no código ou banco
Jobs com retry e fallback — falha silenciosa nunca aceitável
Log de todas as tentativas em notifications_log para auditoria
Webhook LINE validado com assinatura HMAC-SHA256
09 Roadmap de Implementação Ordem de construção — sem pular etapas
Princípio de construção
